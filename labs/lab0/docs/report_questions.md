## Lab0 报告与思考题

1. **本项目中在source code中写死api key的做法是否合理？如果不合理，该怎么改？关于api key提交代码的时候要注意什么？**

2. **LLM 在本 Lab 中的角色**
   - 结合你的实现，LLM 更像是在做：
     - 复杂算法的"黑盒算力"？（例如你把图结构全丢给它，让它自己算路径）
     - 逻辑 / 控制流的"调度 / 推理层"？（例如你自己完成大部分遍历，只让它做少量决策）
     - 代码生成助手？（例如只帮你想出 BFS 写法，但最终运行的仍是传统算法）
   - 请描述你在 Lab 中实际是如何使用 LLM 的，并分析这种用法的优点和局限。

3. **Tool Calling 与 Prompt 工程**
   - 对比 Tool Calling API 和自由文本 JSON 输出，你觉得各有什么优劣？
   - 你在设计 prompt 和 function schema 时遇到了哪些挑战？做了哪些调整？

4. **规模扩展与系统设计**
   - 假设图的节点数扩展到 \(10^5\) 或更大，你觉得本 Lab 中这种"把局部结构编码进 prompt 再让 LLM 算"的方式是否还能正常工作？请从以下角度进行分析：
     - prompt 长度与模型上下文限制；
     - 调用次数与时间开销；
     - 错误率和可验证性。
   - 如果你要在真实系统中加入 LLM 来增强图 / KV 查询能力，你会如何设计整体架构？例如：
     - 哪些部分仍由传统数据结构 / 算法完成；
     - 哪些决策或近似查询可以交给 LLM；
     - 如何在出现错误时降级或回退。
