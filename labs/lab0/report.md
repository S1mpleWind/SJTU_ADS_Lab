# 思考题

1. 写死api的方法不合理，如果代码不留神传出去会泄露；应该放在环境变量里；提交的时候应该注意要删除自己的api-key
2. Q2：
   - LLM，或者说这里应该特指的是*Agent*，是一种逻辑上的调度，只有在厘清逻辑并给定控制流的情况下，他才能很好的发挥出它的“skill”；如果直接丢给他一个需要复杂逻辑 的问题，他往往解决不了
   - 在我的Skills里，我是给了LLM完整的流程，告诉他在关键节点去调用什么样的工具并做决策。这种方法规定了"Agent"运行的轨迹，也是最可控最不容易出错的结果；而它的局限性就是 agent 做不出任何创造性的工作，在遇到复杂的情况/corner case的时候，会出现无法解决的情况
3. Q3
   1. Tool calling 可以借助一些确定的/专业的工具，在完善调用的流程的情况下可以大大提高正确率，降低幻觉，同时也可以获取一些本地的数据；自由文本的输出优势在于更快，同时上下文现在也比 agent memory更加完善
   2. 设计的时候主要是`kv`的时候遇到了一个问题：一开始我告诉llm遇到空的返回就不要记录，但是llm没能判断“空”是什么，把"none"当作一个返回的值；指出这个问题就好了
